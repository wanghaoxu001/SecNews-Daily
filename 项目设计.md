# 项目：网安情报快报系统（SecNews Daily）

## 背景

我是公司网安运营专家，每天需向领导发布一篇经筛选编辑的网安情报 PDF 快报。已有一个使用一年的原型系统，现在要重新构建，做好架构规划。

## 约束

- 单用户系统，用户名 + 密码登录，使用 JWT 认证。用户名密码通过环境变量配置，不建数据库表。前端登录页简洁即可，登录后 token 存 localStorage
- 日均新闻量约 100-300 条
- 部署方式：Docker Compose（backend + frontend + postgres）

### 后端技术栈

- Python 3.12 / FastAPI，全异步
- SQLAlchemy 2.0（async）+ Alembic 迁移 + PostgreSQL 17（启用 pgvector 扩展）
- APScheduler（AsyncIOScheduler）定时任务
- feedparser 解析 RSS，Crawl4AI 抓取正文
- httpx（async）调用 LLM，走 OpenAI 兼容接口，统一封装重试与 token 用量记录
- langdetect 语言检测，jieba 中文分词（用于相似性判断的关键词重叠计算）
- 测试：pytest + pytest-asyncio + httpx（作 TestClient）

### 前端技术栈

- Vue 3（Composition API）+ TypeScript + Vite
- Vue Router 路由，Pinia 状态管理
- Naive UI 组件库
- Axios HTTP 请求
- vuedraggable（基于 SortableJS）实现快报编辑页拖拽排序
- 测试：Vitest + Vue Test Utils

### Git 工作流

- main 分支始终保持可部署状态
- 新功能在 feature/xxx 分支开发，完成后提 PR 合并到 main
- Bug 修复用 fix/xxx 分支
- 提交信息遵循 Conventional Commits 格式（feat: / fix: / refactor: / docs: / test:）
- 合并前须通过全部测试

### 测试要求

- 后端：为每个 API 路由编写集成测试（用测试容器或独立测试库）；为处理流水线各模块编写单元测试，LLM 调用和外部依赖用 mock
- 前端：为包含交互逻辑的组件编写单元测试（勾选缓存、滚动进度、筛选逻辑等）
- 不追求覆盖率数字，但核心业务逻辑必须有测试覆盖

## 核心数据模型（可根据需要调整）

RssSource：name, url, enabled, last_fetched_at

News：rss_source_id(FK), url(UNIQUE), guid, original_title, original_summary, original_content, crawled_content, original_language, title_zh, summary_zh, content_zh, category(枚举), process_status(枚举), process_error, is_similar, similar_to_news_id(FK→News), similar_reason, is_important, importance_reason, importance_strategy, raw_rss_entry(JSON，保存 RSS 条目全部原始字段), published_at, fetched_at, processed_at, embedding(vector，用于语义相似度检索)

Briefing：title, briefing_date, status(draft/published), created_at, updated_at

BriefingItem：briefing_id(FK), news_id(FK), edited_title, edited_summary, category, sort_order —— 创建时 edited_title / edited_summary 从 News 的 title_zh / summary_zh 预填充

LlmConfig：task_type(UNIQUE, 枚举: translate / summarize / classify / similarity / importance / embedding), base_url(可空), api_key(可空), model_name, temperature, max_tokens, system_prompt —— 另有一条 task_type=default 的记录存放全局默认 base_url 和 api_key，各任务配置为空时自动继承全局值

TaskConfig：task_name(UNIQUE), enabled, cron_expression, last_run_at, last_run_status

ImportanceExample：category, title, summary

ProcessingConfig：config_key(UNIQUE), config_value, description —— 存放各模块可调参数（摘要字数下限、相似性权重/阈值等）

枚举值：

- ProcessStatus: pending → processing → processed → similarity_checked → completed / failed
- NewsCategory: 金融业网络安全事件 / 重大网络安全事件 / 重大数据泄露事件 / 重大漏洞风险提示 / 其他

## 页面设计

四个页面：登录页 + 三个业务页面（今日新闻、快报、管理），业务页面间通过顶部导航切换。未登录时重定向到登录页。

### 今日新闻

1. 新闻卡片列表，卡片显示：中文标题、中文摘要（截断到固定最大字数）、分类标签、发布时间、来源 RSS 名称、疑似重复标记（橙色，如有）、建议收录标记（金色，如有）
2. 默认筛选：时间起点 = 上一份快报的创建时间（无历史快报则取过去 24 小时）；分类 = 四大分类，排除"其他"。筛选条件可手动调整
3. 支持勾选新闻，勾选状态存 localStorage，以日期为 key，仅保留当天
4. 点击卡片主体弹出侧边详情抽屉，展示全部字段及原文链接
5. 向下滚动时顶部固定显示浏览进度条 + "生成快报"按钮
6. 点击"生成快报"→ 用勾选的新闻创建草稿快报（BriefingItem 从 News 预填充标题和摘要）→ 跳转快报编辑页

### 快报

1. 列表展示历史快报：日期、标题、状态（草稿/已发布）
2. 点击进入快报编辑页

### 快报编辑页

1. 按分类分组展示条目
2. 每条可编辑标题、摘要
3. 下拉切换分类（自动移到对应分组）
4. 拖拽排序
5. 保存按钮
6. "导出 PDF"按钮置灰，标注"即将推出"

### 管理页面（Tab 组织）

1. LLM 配置：顶部配置全局默认 base_url 和 api_key；下方表格为 6 种任务（translate / summarize / classify / similarity / importance / embedding）分别配置 model_name、temperature、max_tokens、system_prompt，以及可选的独立 base_url / api_key（留空则继承全局）
2. RSS 源管理：增删改，启用/禁用切换
3. 定时任务：配置流水线的 cron 表达式、启用/禁用、手动触发按钮、上次运行状态显示
4. 处理参数：以 key-value 表格管理各模块可调参数
5. 重要性样本：按分类 Tab 管理典型重要新闻样本（标题 + 摘要），支持增删改，支持 JSON 格式批量导入（格式：`[{"category": "...", "title": "...", "summary": "..."}]`）
6. 重要性策略：选择当前启用的判断子模块（当前仅子模块 A）

## 后端处理流水线

四个模块按序串联：RSS 抓取 → 初步处理 → 相似性判断 → 重要性判断。定时任务或手动触发时按序执行，每个模块也支持单独触发。

### 模块 1：RSS 抓取

遍历启用的 RSS 源，只抓取该源数据库中最新一条之后的条目。按 URL 去重，存入 News 表（status=pending，raw_rss_entry 保存完整原始字段）。单源失败不影响其他源。

### 模块 2：新闻初步处理

对每条 pending 新闻执行：

获取中文摘要（二选一路径）：

- 路径 A（有可用摘要）：RSS 条目存在摘要字段（summary / description 等取首个非空）且字数 ≥ 下限（可配置）→ 非中文则 LLM 翻译 → 得到中文摘要
- 路径 B（需从正文生成）：摘要不存在或字数不足 → 取正文（RSS 自带 content / content_encoded 优先，否则 Crawl4AI 抓取；抓取失败则降级：有短摘要走路径 A，无则标记失败）→ 非中文则翻译正文 → LLM 对正文生成摘要

获取中文标题：非中文则 LLM 翻译

分类：LLM 根据中文标题 + 中文摘要归入五类之一

全部完成 → status=processed；任何步骤出错 → status=failed + 记录原因。处理应具备幂等性，失败重跑可从断点继续。

### 模块 3：相似性判断

对每条 processed 新闻，与近期已发布快报中的新闻去重：

1. 取过去 N 天（可配置，默认 14）已发布快报中的新闻，筛选同分类子集作为候选

2. 初筛算法

   （加权求和，权重/阈值均可在管理页面配置）：

   - 实体匹配：提取组织名、CVE 编号、产品名等，Jaccard 系数 + CVE 完全匹配额外加分
   - 语义相似度：embedding 向量（调用 embedding 接口生成，存入 News.embedding 字段）的余弦相似度，通过 pgvector 检索
   - 关键词重叠：分词去停用词后 TF-IDF 余弦相似度

3. 综合分超过阈值的取 Top-K 送入 LLM 做最终判定：是否属于同一事件

4. 判定重复则标记 is_similar=true，记录关联新闻 ID 和原因

### 模块 4：重要性判断

采用策略模式，支持多种子模块，在管理页面切换启用哪种。

子模块 A（基于典型样本的 LLM 判断）：

1. 取管理页面维护的同分类典型样本集合
2. 构建 prompt：样本集合置于前部（利用 LLM prompt caching，同分类新闻共享前缀以节约耗时和费用），待判断新闻置于末尾
3. LLM 返回是否重要 + 理由，写入 News 表
